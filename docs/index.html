THE QUEUE ALWAYS GROWS

Finished downloads should not become permanent furniture.

A busy Transmission instance quietly accumulates completed torrents, fills storage, and makes it harder to see what actually needs attention. Transmission Cleaner gives that work a schedule, a policy, and a clear audit trail.

Connect one or more servers, define the rules, and let a small container handle the routine.

HOW IT DECIDES

Four gates. One deliberate deletion.

Every candidate passes the same sequence. If a gate fails, the torrent stays put.

  1. 01

    Seeding or stopped

    Only torrents in a removable state enter the policy.

  2. 02

    100% complete

    Incomplete downloads are never touched.

  3. 03

    Old enough

    The torrent must be at least DAYS_TO_WAIT days old.

  4. 04

    Ratio threshold

    When enabled, upload ratio must meet MIN_RATIO.

The default policy waits 10 days and leaves the ratio check disabled.

COMMON QUESTIONS

Good housekeeping starts with clear boundaries.

What do I need to run it?

Docker 24+ (or Podman with Docker Compose support), plus network access from the container to each Transmission RPC endpoint. The default RPC port is 9091.

Does dry-run mode delete anything?

No. It fetches torrents and marks what would be deleted without deleting torrents or sending notifications. Dry runs are not stored in cleanup history.

How can it notify me?

Telegram, Resend, and SMTP can each be enabled independently. Notifications are sent in parallel and are configurable through environment variables.

Where does cleanup history live?

Effective runs are stored in the SQLite database at /config/cleanups.db and can be searched from /history. Legacy .eml notifications can be imported with the included script.

Can I integrate it with another tool?

Yes. The REST API exposes /api/status and /api/torrents for scripting or external integrations, and /api/status also powers the Docker health check.
